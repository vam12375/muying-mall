  二、架构设计评估

  1. 整体分层 — 合理但有改进空间

  项目采用经典的 Controller → Service → Mapper 三层架构，辅以 entity、dto
  、enums、event、statemachine、consumer、listener、util、config
  等包，结构清晰。

  问题 (SOLID - 单一职责违反):
  - AdminController.java 达到 43KB，承载了过多的管理端逻辑
  - OrderServiceImpl.java 达到 79KB，职责过于庞大
  - CommentServiceImpl.java 达到 56KB
  - PointsServiceImpl.java 达到 66KB
  - ProductServiceImpl.java 达到 44KB
  - ProductSearchServiceImpl.java 达到 47KB

  这些"上帝类"违反了SRP原则，导致可维护性下降、测试困难。

  2. 单体架构 — 当前阶段合理

  对于母婴商城的业务规模，单体架构是合适的选择(YAGNI)。但需关注随业务增  
  的拆分路径。

  3. 状态机模式 — 设计亮点

  statemachine 包实现了订单、支付、退款三个状态机，是良好的设计模式应用  

  4. 事件驱动 — 设计亮点

  使用 Spring Event + RabbitMQ 双层事件驱动，解耦了订单完成、积分变更、  
  流提醒等业务，符合开闭原则(OCP)。

  三、严重问题 (需优先修复)

  P0: 全局异常处理器掩盖错误

  // GlobalExceptionHandler.java:70-75
  @ExceptionHandler(Exception.class)
  public Result<Void> handleException(Exception e) {
      log.error("系统异常: {}", e.getMessage());  // 没有打印堆栈        
      return Result.success(null, "系统繁忙，请稍后再试");  //
  吞掉异常返回200
  }

  问题：
  - 所有未知异常都返回 success=true，注释说"避免JMeter将其判定为错误"。  
  是为了压测指标好看而掩盖真实问题
  - 只记录 e.getMessage() 没有完整堆栈，出问题时无法定位
  - 上游监控系统(如 Prometheus)无法捕获真实错误率

  P0: CacheRefreshService 大量使用 keys() 命令

  // CacheRefreshService.java:92
  Set<String> keys = redisTemplate.keys(pattern);

  KEYS 命令在生产环境Redis中是 O(N) 操作，会阻塞Redis线程。高并发下可能  
  接导致Redis响应超时，影响所有依赖Redis的服务。应改用 SCAN 命令。       

  P0: HikariCP 连接池配置过大

  # application-performance.yml
  hikari:
    minimum-idle: 150
    maximum-pool-size: 600

  MySQL 单实例默认 max_connections=151。配置600个连接池几乎必然导致连接  
  尽或MySQL拒绝连接。对于单体应用，建议 maximum-pool-size 设为 CPU核心数 
  * 2 + 磁盘数，通常 20-50 即可满足高并发需求。

  P1: Redis 连接池配置过大

  lettuce:
    pool:
      max-active: 512
      min-idle: 128

  单机Redis难以承受512并发连接。且 min-idle: 128
  意味着空闲时也维持128个连接，浪费资源。建议 max-active:
  32-64，min-idle: 8-16。

  P1: 分布式锁释放不安全

  // CacheProtectionUtil.java:199-203
  if (lockAcquired) {
      redisUtil.del(lockKey);  // 直接删除，没有验证锁的持有者
  }

  使用 del 直接删除锁，没有比较持有者标识。如果锁过期后被其他线程获取，  
  前线程可能误删他人的锁。项目中已有 RedisDistributedLock
  使用Lua脚本正确实现，但 CacheProtectionUtil 没有使用它。

  P1: TccConfig 序列化器注入问题

  // TccConfig.java:31
  public RedisTemplate<String, Object> tccRedisTemplate(
      RedisConnectionFactory connectionFactory,
      ObjectMapper objectMapper)  //
  注入的是HTTP用的ObjectMapper，不是Redis专用的

  这里注入的 ObjectMapper 是Spring默认的HTTP序列化器(不带类型信息)，与   
  RedisConfig 中的 redisObjectMapper(带类型信息)
  不一致，会导致TCC事务数据反序列化失败。

  四、性能问题分析

  1. N+1 查询仍存在

  OrderServiceImpl.createOrder()
  中虽然对商品做了批量查询优化，但对SKU仍然是逐个查询：

  // OrderServiceImpl.java:215
  ProductSku sku = productSkuService.getById(cart.getSkuId());  //       
  循环内逐个查询

  2. 物流定时任务每分钟全表扫描

  // LogisticsScheduledTask.java:36
  @Scheduled(cron = "0 */1 * * * *")  // 每分钟执行一次

  每分钟扫描所有运输中的物流记录，如果数据量增大会成为性能瓶颈。建议增加 
  索引 + 分页处理或使用消息延迟队列替代轮询。

  3. 配置冲突

  application.yml 和 application-performance.yml 存在重复配置(如Redis连  
  池、MyBatis-Plus、日志等)。spring.profiles.active=performance
  会覆盖主配置，但重复维护增加出错风险。

  4. PageHelper 与 MyBatis-Plus 分页插件冲突

  同时引入了 pagehelper-spring-boot-starter 和 MyBatis-Plus
  自带的分页插件(通过 MybatisPlusConfig.java)，两套分页可能产生冲突。建  
  保留一个(推荐MyBatis-Plus自带的)。

  5. Elasticsearch 刷新间隔

  refresh-interval: 30s  # 30秒刷新

  30秒的刷新间隔意味着数据写入后最多30秒才能被搜索到。对于商品搜索场景这 
  是可接受的，但需要确认业务是否接受这个延迟。

  五、安全问题

  1. SecurityConfig 存在过度放行

  .requestMatchers("/refund/**").permitAll()  // 退款接口全部放行        

  退款相关接口对所有人开放，可能导致未授权的退款信息泄露。

  2. Redis ObjectMapper 使用 LaissezFaireSubTypeValidator

  // RedisConfig.java:56
  objectMapper.activateDefaultTyping(LaissezFaireSubTypeValidator.instanc
  e, ...)

  LaissezFaireSubTypeValidator
  允许所有类型的反序列化，存在反序列化漏洞风险。应使用白名单机制。       

  3. JCenter 仓库已废弃

  <repository>
      <id>jcenter</id>
      <url>https://jcenter.bintray.com</url>
  </repository>

  JCenter 于 2021 年已停止服务，应移除此仓库配置。

  ---
  六、代码质量问题

  1. 测试覆盖率为零

  src/test 目录下没有任何测试文件，完全依赖手动测试。对于包含支付、退款  
  库存扣减等关键业务逻辑的系统，这是高风险的。

  2. DTO / Entity 混用

  部分Controller直接返回Entity对象，暴露了数据库字段(如
  is_deleted、password 等)。应使用DTO严格隔离。

  3. 缓存Key管理混乱

  CacheRefreshService 中硬编码了多种格式的缓存Key(order:detail:、order_de
  tail_、muying:order:detail: 等)，说明缓存Key命名不统一。

  4. 依赖版本不一致

  <poi>5.2.4</poi>
  <poi-ooxml>5.4.0</poi-ooxml>

  Apache POI 的两个模块版本不一致，可能导致运行时兼容性问题。

  七、设计亮点

  1. 虚拟线程启用：充分利用Java 21虚拟线程，简化并发模型
  2. Lua脚本秒杀：使用Redis Lua脚本保证库存扣减的原子性，包含用户去重逻  
  3. 缓存防护体系：实现了穿透保护、击穿互斥锁、雪崩随机过期的完整三件套  
  4. TCC分布式事务：订单和支付使用TCC模式保证最终一致性
  5. 消息队列降级：RabbitMQ不可用时自动降级为同步处理
  6. 状态机设计：订单/支付/退款使用状态机管理生命周期
  7. 分布式锁：基于Redis + Lua的正确实现(虽然未被统一使用)
  8. 配置分离：敏感信息通过 application-private.yml 分离
