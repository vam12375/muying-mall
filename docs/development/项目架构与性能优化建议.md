# 母婴商城项目架构与性能优化建议书

## 1. 文档目的

本文档用于对当前母婴商城后端项目进行系统性评审，给出可执行的优化建议，目标如下：

- 评估当前架构设计合理性和风险点。
- 识别性能瓶颈与稳定性隐患。
- 制定分阶段优化路线（P0/P1/P2）。
- 提供可落地的技术方案、验收指标与推进计划。

---

## 2. 项目现状概览

从项目结构看，系统已具备较完整的电商后端能力，采用 Spring Boot 单体架构，具备以下特征：

- 分层明确：Controller、DTO、Config、AOP、Consumer 等职责基本清晰。
- 业务覆盖完整：用户、商品、订单、支付、退款、积分、钱包、评论、营销、后台管理等。
- 中间件能力较全面：Redis、RabbitMQ、Elasticsearch、异步线程池、WebSocket。
- 有一定可观测与治理基础：日志切面、性能监控切面、系统监控接口、统一响应与异常处理。

**总体评价**：架构方向合理，功能完整度较高，已具备中大型业务系统雏形。

---

## 3. 架构合理性评估

### 3.1 优势

1. **单体架构下的模块化较好**
   - 包结构清晰，后续做模块边界强化或拆分服务有基础。

2. **关键链路已异步化**
   - 使用 RabbitMQ 处理订单、支付、超时、死信，具备削峰填谷能力。

3. **性能优化意识较强**
   - 引入缓存切面、缓存常量、秒杀相关配置，说明已考虑热点场景。

4. **配置体系完善**
   - 配置类分工较细，减少配置耦合风险。

5. **业务场景完整度高**
   - 覆盖交易、资产、履约与内容生态，符合真实电商复杂度。

### 3.2 主要风险

1. **超大控制器导致维护风险上升**
   - 如 `PaymentController`、`WalletPaymentController`、`AdminController`。
   - 风险：逻辑耦合、事务边界不清晰、回归测试成本高、多人协作冲突大。

2. **交易一致性复杂度高**
   - 订单/支付/退款/钱包/积分存在跨模块状态同步与补偿问题。
   - 缺乏统一幂等标准时，易出现重复处理或状态不一致。

3. **缓存一致性与热点保护可能不足**
   - 若未统一 TTL、随机过期、空值缓存、防击穿，热点高峰时易抖动。

4. **消息系统运维风险**
   - 常见问题包括重试风暴、死信堆积、消费延迟、消费者顺序性问题。

5. **数据库可能成为最终瓶颈**
   - 电商核心链路最终受限于索引设计、SQL 写法、分页策略与锁冲突。

## 4. 性能现状与关键链路分析

### 4.1 商品查询/搜索链路

- 已有 ES 与缓存基础，具备横向扩展潜力。
- 重点风险：
  - 索引更新延迟导致数据可见性窗口。
  - 列表页关联查询导致 N+1 问题。
  - 热门商品缓存预热与失效策略不足。

### 4.2 下单链路

- 涉及库存、价格、营销、运费、订单主从表写入。
- 重点风险：
  - 单事务过大，导致锁持有时间长。
  - 库存/优惠券/账户并发更新冲突。
  - 重试与超时机制不完善时出现重复订单。

### 4.3 支付链路

- 包含第三方支付与钱包支付，属于最关键风险路径。
- 重点风险：
  - 回调重复通知幂等不足。
  - 订单状态与支付状态不同步。
  - MQ 与数据库双写窗口导致一致性问题。

### 4.4 退款链路

- 涉及订单逆向状态、资金退回、积分/优惠券回滚。
- 重点风险：
  - 并发退款重复执行。
  - 部分成功后补偿不完整。
  - 对账与审计链路不完善。

### 4.5 秒杀链路

- 有独立配置，架构方向正确。
- 重点风险：
  - 入口限流不足导致服务雪崩。
  - Redis 预扣与 DB 最终库存一致性。
  - 下游 MQ 堆积导致响应延迟。

### 4.6 后台统计链路

- 若实时聚合直扫交易大表，易影响线上主业务。
- 建议采用汇总表、异步计算与分层缓存。

## 5. 核心优化建议（分优先级）

## 5.1 P0：稳定性与一致性（必须优先）

### 5.1.1 建立统一幂等规范

建议定义幂等键：

`idempotent_key = business_type + business_id + action + request_id`

适用场景：

- 支付回调
- 退款回调
- MQ 消费
- 关键状态变更接口

落地要求：

- 幂等记录入库（唯一索引约束）。
- 幂等命中直接返回已处理结果。
- 所有资金操作必须“可重试、不可重复生效”。

### 5.1.2 收敛状态机与事务边界

- 为订单、支付、退款、钱包建立状态迁移白名单。
- 更新语句增加状态前置条件（CAS 风格）：
  - 示例：`update ... where id = ? and status = OLD_STATUS`
- 强制禁止跨层直接改状态。

### 5.1.3 MQ 可靠性治理

- 统一重试策略：固定次数 + 指数退避。
- 死信队列处理流程标准化（告警、人工介入、自动回放策略）。
- 消费者增加三类指标：失败率、重试率、消费延迟。

### 5.1.4 慢 SQL 治理机制化

- 开启慢 SQL 日志（建议阈值 200ms 起步）。
- 每周治理 Top N 慢 SQL。
- 重点优化：索引覆盖、排序字段索引、避免大偏移分页、减少回表。

---

## 5.2 P1：性能提升（高收益）

### 5.2.1 缓存体系标准化

- 统一 key 规范：`biz:entity:id[:field]`
- TTL 分层：短/中/长，并加随机抖动。
- 增加空值缓存防穿透。
- 热点 key 增加互斥锁或逻辑过期策略。
- 明确 Cache Aside 读写流程与失效策略。

### 5.2.2 连接池与线程池调优

- HikariCP 参数按实例规格与并发模型调整。
- 异步线程池按场景拆分（IO 密集、CPU 密集、通知类任务）。
- 明确队列容量、拒绝策略、告警阈值。

### 5.2.3 热点接口限流与降级

- 秒杀、支付查询、商品详情等接口增加限流。
- 下游依赖超时时快速失败并返回降级结果。
- 防止高峰期级联故障。

### 5.2.4 查询与分页优化

- 大分页改为基于主键/时间游标翻页。
- 高频列表接口使用覆盖索引与必要字段查询。
- 减少多表深度 join，必要时引入冗余字段与汇总表。

---

## 5.3 P2：架构演进（中长期）

### 5.3.1 先做“模块强边界”再考虑微服务

推荐领域划分：

- 交易域（订单、支付、退款）
- 商品域（SPU/SKU、库存、搜索）
- 用户资产域（钱包、积分、优惠券）

实践方式：

- Controller 轻量化。
- 应用服务负责编排。
- 领域服务实现业务规则。
- 仓储层专注数据访问。

### 5.3.2 关键事件引入 Outbox 模式

- 使用本地事务写业务数据与事件表。
- 异步投递事件到 MQ。
- 保证“业务提交成功 = 事件最终可达”。

### 5.3.3 完整可观测性建设

核心指标建议：

- 接口：QPS、P95/P99、错误率
- 缓存：命中率、穿透率、热点 key
- MQ：堆积量、延迟、失败重试
- DB：慢 SQL 数、锁等待、连接池使用率

### 5.3.4 压测与容量模型

至少建设三套压测场景：

- 常态流量
- 大促峰值
- 秒杀极限

输出结果包括：

- 单机极限吞吐
- 集群扩容曲线
- 首要瓶颈迁移路径

## 6. 代码结构优化建议（针对当前项目）

以下文件建议优先进行“减重重构”：

- `src/main/java/com/muyingmall/controller/common/PaymentController.java`
- `src/main/java/com/muyingmall/controller/user/WalletPaymentController.java`
- `src/main/java/com/muyingmall/controller/admin/AdminController.java`
- `src/main/java/com/muyingmall/config/RabbitMQConfig.java`

优化目标：

- 控制器仅做参数校验与路由转发。
- 复杂业务下沉到应用服务与领域服务。
- 抽离通用支付/退款/消息处理组件。
- 统一错误码与异常语义。

---

## 7. 30 天执行计划（建议）

### 第 1 周：稳定性兜底

- 梳理支付、退款、订单状态机。
- 落地幂等表与唯一约束。
- 核心消费者幂等改造。
- 建立死信处理 SOP。

验收指标：

- 重复回调不再导致重复入账。
- 消费失败可重试且无重复副作用。

### 第 2 周：数据库与缓存治理

- 开启慢 SQL 日志并优化 Top 20。
- 完成缓存 key 与 TTL 规范化。
- 增加热点保护与空值缓存。

验收指标：

- 核心接口 P95 降低 20% 以上（目标值可按当前基线调整）。
- 缓存命中率达到目标阈值（如 > 85%）。

### 第 3 周：限流降级与线程池优化

- 热点接口加限流与熔断。
- 调整线程池与连接池参数。
- 建立高峰告警规则。

验收指标：

- 压测峰值时错误率明显下降。
- 队列堆积恢复时间可控。

### 第 4 周：模块边界与可观测性建设

- 完成核心大控制器拆分一期。
- 接入统一指标看板。
- 输出容量评估报告与下一阶段计划。

验收指标：

- 关键模块代码复杂度下降。
- 排障时间（MTTR）显著缩短。

---

## 8. 建议的验收指标体系（可直接采用）

### 稳定性

- 交易链路重复处理率：接近 0
- 死信消息处理及时率：> 99%
- 线上 Sev1/Sev2 事故次数：环比下降

### 性能

- 核心接口 P95/P99：持续下降
- 数据库慢 SQL 数量：持续下降
- 缓存命中率：稳定在目标区间

### 可维护性

- 大文件数量（>2000 行）：持续减少
- 场景回归耗时：下降
- 发布失败回滚率：下降

---

## 9. 风险与回滚策略

### 9.1 优化过程风险

- 幂等改造可能影响既有重试流程。
- 索引调整可能影响写入性能。
- 限流策略过严可能影响用户体验。

### 9.2 回滚策略

- 所有关键开关支持配置化灰度。
- 新增索引先在影子环境压测后上线。
- 限流和熔断阈值先保守后收紧。
- 大改造按模块分批发布，确保可回退。

---

## 10. 结论

当前系统的架构方向整体正确，功能完整度高，具备生产化基础。下一阶段不应再以“堆功能”为主，而应转向以下三点：

1. **一致性可证明**：幂等、状态机、补偿机制闭环。
2. **性能可量化**：基于基线、压测和指标持续优化。
3. **故障可恢复**：可观测、告警、回滚、应急流程完备。

建议按本文 P0/P1/P2 路线推进，可在 1-2 个迭代内显著提升系统稳定性与性能上限。
