# 顺丰沙箱 API 接入方案

## 1. 文档目的

本文档用于指导母婴商城项目接入顺丰丰桥开放平台沙箱环境，并按照未来可平滑切换生产环境的思路完成系统设计。

本文档覆盖以下内容：

- 当前项目现状与差距分析
- 顺丰沙箱接入目标与范围
- 总体架构设计
- 配置设计
- 数据模型与数据库扩展建议
- 后端改造清单
- 同步策略设计：先轮询，后推送
- 前端优雅展示方案
- 联调与测试清单
- 上线与切换建议
- 风险与注意事项

本文档定位为设计文档，不等同于最终实现代码。接口字段、服务码、签名规则、返回码解释必须以顺丰开放平台最新官方文档为准。

## 2. 背景

当前项目已经具备完整的订单、发货、物流记录、物流轨迹和地图展示能力，但现有物流链路主要是本地模拟逻辑，不是真实第三方物流接入。

当前项目中与物流相关的能力包括：

- 订单发货后创建 `logistics` 记录
- 根据物流公司生成本地物流单号
- 根据物流公司模板生成标准轨迹
- 使用高德地图生成演示性质的路径轨迹
- 定时任务根据本地轨迹自动推进物流状态
- 前台和后台提供物流查询接口

这些能力适合演示和本地业务闭环，但不适合直接承接真实顺丰物流数据。

## 3. 接入目标

### 3.1 业务目标

- 在管理员发货时，对接顺丰沙箱进行真实下单
- 获取顺丰返回的真实运单号并保存到本地
- 定时查询顺丰路由信息并写入本地物流轨迹
- 为前端提供真实物流时间线
- 在地图上对真实节点做“城市跳点式”的辅助展示
- 代码结构支持未来从沙箱平滑切换到生产环境
- 代码结构支持未来从轮询模式升级到顺丰回调推送模式

### 3.2 展示目标

本次目标不是复刻“精准运输路径”，而是做“真实时间线 + 地图辅助展示”。

展示层的核心原则如下：

- 时间线使用顺丰真实节点数据
- 地图只做空间变化增强，不伪造真实行驶路线
- 页面要优雅、清晰、可信
- 支持节点高亮、当前状态高亮、起终点摘要和城市跳点连线

### 3.3 非目标

本次设计不做以下内容：

- 不实现生产环境正式开通流程
- 不处理顺丰电子面单打印细节
- 不实现多物流平台统一接入的完整产品化后台
- 不实现高精度运输路径回放
- 不强依赖公网回调地址

## 4. 官方能力边界

根据顺丰开放平台官方资料，顺丰提供的能力主要是：

- 下单
- 查路由
- 路由注册
- 路由推送

顺丰返回的核心物流节点信息本质上是文本化事件，例如：

- 节点时间
- 节点地点
- 节点描述
- 节点操作码

顺丰开放平台并不直接提供适用于本项目前端的“地图轨迹展示组件”，也不保证所有节点都带有可直接用于地图绘制的经纬度。因此：

- 时间线可以直接基于真实数据构建
- 地图展示需要项目侧自行增强
- 地图增强应以“地点转坐标”和“城市级跳点展示”为主

## 5. 当前项目现状分析

### 5.1 已具备的基础能力

- Spring Boot 3.3 + Java 21 技术栈稳定
- 已有 `RestTemplate`、`ObjectMapper` 等 HTTP 与序列化基础设施
- 已有物流域模型：`Logistics`、`LogisticsTrack`、`LogisticsCompany`
- 已有后台发货入口与前后台物流查询接口
- 已有高德地图能力，可复用作地理编码和地图展示
- 已有异步事件机制与定时任务机制

### 5.2 当前阻塞点

当前项目不适合直接把顺丰接进现有逻辑，主要问题如下：

1. 物流单号是本地生成的，不是真实第三方运单号
2. “顺丰轨迹”目前只是本地模板，不是真实顺丰路由
3. 管理员发货链路里会先建本地物流记录，再生成本地轨迹
4. 异步事件链路里还存在自动创建物流记录的逻辑
5. 定时任务会根据本地轨迹自动推进物流状态
6. 当前配置体系没有顺丰接入专用配置
7. 当前前端地图轨迹依赖经纬度，而顺丰节点未必能直接提供经纬度

### 5.3 结论

项目具备接入顺丰沙箱的工程基础，但必须把“模拟物流”和“真实顺丰物流”进行逻辑隔离。推荐保留现有模拟能力，但只对非第三方物流或演示物流使用。

## 6. 总体方案

### 6.1 方案结论

采用以下方案：

- 接入模式：顺丰沙箱优先，结构按未来生产切换设计
- 同步模式：当前阶段使用主动轮询，未来升级为推送为主、轮询兜底
- 展示模式：真实时间线为主，城市跳点地图为辅

### 6.2 分层设计

系统拆分为以下五层：

#### 1）物流接入抽象层

定义统一接口，屏蔽物流供应商差异，建议能力包括：

- 创建运单
- 查询路由
- 注册路由推送
- 校验推送签名

建议接口名：

- `ExpressProviderClient`
- `ExpressRouteSyncService`

#### 2）顺丰实现层

专门负责顺丰协议细节，包括：

- 公共请求参数拼装
- `msgDigest` 签名生成
- 请求发送
- 顺丰响应解析
- 异常码映射
- 沙箱与生产地址切换

#### 3）物流标准化层

将顺丰原始数据转换为项目统一的物流模型：

- 物流主记录
- 物流节点
- 当前物流状态
- 前端展示摘要
- 地图展示点

#### 4）物流同步层

负责把顺丰数据同步到本地：

- 轮询任务扫描待同步运单
- 查询顺丰路由
- 幂等落库
- 更新物流主状态
- 触发地点转坐标增强

#### 5）物流展示层

前端只依赖项目自身接口，不直接消费顺丰报文：

- 物流摘要接口
- 时间线接口
- 地图点位接口

## 7. 总体业务流

### 7.1 管理员发货流程

```text
管理员发货
  -> 校验订单状态、物流公司、幂等性
  -> 如果选择顺丰
       -> 调顺丰沙箱下单
       -> 获取真实运单号
       -> 保存订单发货信息
       -> 保存本地物流主记录
       -> 标记该单为第三方顺丰物流
       -> 可选：立即触发一次首轮路由同步
  -> 返回发货成功
```

### 7.2 路由同步流程

```text
定时任务扫描待同步顺丰运单
  -> 调顺丰查路由
  -> 路由节点标准化
  -> 幂等写入 logistics_track
  -> 更新 logistics 当前状态
  -> 对地点做坐标补全或缓存命中
  -> 生成地图展示点
```

### 7.3 展示流程

```text
前端查询物流详情
  -> 后端返回物流摘要 + 时间线 + 地图点位
  -> 前端展示当前状态卡片
  -> 时间线展示真实节点
  -> 地图展示城市跳点、当前节点、起终点
```

## 8. 配置设计

### 8.1 配置原则

- 接入密钥不放数据库，统一放私有配置文件或环境变量
- 沙箱与生产通过配置切换，不改业务代码
- 轮询与推送模式可以通过配置开关控制

### 8.2 建议配置项

建议新增以下配置：

```yaml
sf:
  express:
    enabled: true
    sandbox-enabled: true
    base-url: https://sfapi-sbox.sf-express.com/std/service
    client-code: your_client_code
    check-word: your_check_word
    monthly-account: your_monthly_account
    sync-mode: polling
    route:
      polling-enabled: true
      polling-cron: 0 */10 * * * *
      max-batch-size: 100
      max-route-days: 30
    callback:
      enabled: false
      notify-url: https://your-domain/api/sf/route/callback
    geocode:
      enabled: true
      cache-days: 30
      fallback-city-level: true
```

### 8.3 配置建议

- `base-url` 按环境切换
- `client-code`、`check-word` 必须放私有配置
- `sync-mode` 初始值为 `polling`
- `callback.enabled` 第一阶段保持 `false`

## 9. 数据模型设计

### 9.1 复用现有表的原则

优先复用现有表：

- `order`
- `logistics`
- `logistics_track`
- `logistics_company`

在现有表上做增强，避免新建过多核心业务表。

### 9.2 `logistics` 表扩展建议

建议增加以下字段：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| provider_code | varchar(32) | 物流供应商代码，例：`SF` |
| provider_type | varchar(32) | 供应商类型，例：`THIRD_PARTY`、`LOCAL_SIMULATION` |
| provider_order_no | varchar(64) | 顺丰下单返回的第三方订单号或业务单号 |
| provider_waybill_no | varchar(64) | 顺丰真实运单号，建议与 `tracking_no` 同步 |
| provider_status | varchar(32) | 顺丰标准化后的状态 |
| route_sync_mode | varchar(16) | `POLLING` 或 `PUSH` |
| route_sync_status | varchar(16) | `PENDING`、`SYNCING`、`SUCCESS`、`FAILED` |
| last_route_sync_time | datetime | 最近一次路由同步时间 |
| next_route_sync_time | datetime | 下一次建议同步时间 |
| provider_payload_json | json | 最近一次顺丰主响应原文 |
| current_node_time | datetime | 当前最新节点时间 |
| current_node_location | varchar(255) | 当前最新节点地点 |
| current_node_remark | varchar(1000) | 当前最新节点说明 |

说明：

- `tracking_no` 继续保留，作为本项目统一运单号字段
- 对顺丰单来说，`tracking_no` 与 `provider_waybill_no` 值相同
- `provider_*` 字段用于隔离第三方平台信息

### 9.3 `logistics_track` 表扩展建议

建议增加以下字段：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| source_type | varchar(32) | 来源类型，例：`SF_API`、`LOCAL_TEMPLATE` |
| source_event_id | varchar(128) | 外部事件唯一标识，用于幂等 |
| event_code | varchar(64) | 顺丰节点操作码 |
| event_time | datetime | 外部原始节点时间 |
| event_city | varchar(128) | 标准化城市名 |
| geocode_status | varchar(16) | `PENDING`、`SUCCESS`、`FAILED` |
| raw_payload_json | json | 外部节点原始报文 |

### 9.4 可选新表

如果需要增强可观测性，建议新增：

#### `logistics_sync_log`

用于记录每次同步任务：

- 运单号
- 同步时间
- 请求类型
- 请求结果
- 节点新增数
- 错误信息

#### `logistics_geo_cache`

用于缓存地点文本到坐标的映射：

- 供应商
- 原始地点文本
- 标准化地点文本
- 城市名
- 经度
- 纬度
- 置信度
- 最近命中时间

职责：

- 读取顺丰配置

- 管理轮询/推送开关

### 10.2 新增顺丰 DTO

建议新增包：

- `com.muyingmall.dto.sf`

建议拆分：

- 请求 DTO
- 响应 DTO
- 路由节点 DTO

### 10.3 新增接入抽象

建议新增接口：

```java
public interface ExpressProviderClient {
```

建议新增：

- `SfExpressClient`
- `SfSignatureService`

- `SfResponseParser`

职责拆分如下：

- `SfExpressClient`：统一调顺丰

- `SfRequestFactory`：构建顺丰请求

### 10.5 新增标准化服务

建议新增：

- `ExpressLogisticsMapperService`

- 外部报文转内部模型
- 路由节点标准化

### 10.6 新增同步任务

建议新增：

- `SfRoutePollingTask`

职责：

- 扫描待同步顺丰单
- 拉取路由
- 幂等落库
- 更新状态

### 10.7 未来新增回调入口

建议预留：

- `SfRouteCallbackController`

第一阶段可以先只建空能力或不启用路由映射，第二阶段上线推送时启用。

## 11. 对现有代码的改造建议

### 11.1 管理员发货入口改造

当前 `AdminOrderController` 的发货逻辑会：

- 本地生成运单号
- 本地创建物流记录
- 本地生成轨迹

改造后建议：

1. 先判断物流公司是否为顺丰
2. 如果是顺丰：
   - 先调顺丰下单
   - 获取真实运单号
   - 再保存本地物流主记录
   - 不再生成本地标准轨迹和高德插值轨迹
3. 如果不是顺丰：
   - 保持现有本地物流逻辑

### 11.2 `OrderServiceImpl.shipOrder` 改造

该方法应只负责订单状态变更，不应再承担：

- 生成第三方物流单号
- 发布误导性的“支付后自动创建物流”事件

建议：

- 将第三方物流创建放到专门的发货服务中
- `shipOrder` 只接收已准备好的运单结果

### 11.3 `OrderPaidEventListener` 改造

当前监听器内存在自动创建物流的逻辑，并且写死顺丰样式单号，这不适合真实第三方物流。

建议：

- 保留监听器能力，但只用于本地模拟物流
- 对第三方物流单直接禁用该监听逻辑
- 增加条件判断：若订单标记为第三方物流，则跳过自动创建

### 11.4 `LogisticsScheduledTask` 改造

当前任务会根据本地轨迹自动推进状态，这对真实顺丰物流是有害的。

建议：

- 对 `provider_type = THIRD_PARTY` 的单据禁用本地推进
- 新增顺丰专用轮询任务
- 本地模拟物流任务与第三方同步任务分离

### 11.5 `LogisticsController` 改造

现有前台查询接口只按本地 `trackingNo` 查库，`companyCode` 参数没有真正参与第三方查询逻辑。

建议：

- 前台只查本地统一结果
- 不在前端接口层直接请求顺丰
- 增加统一响应结构，包含：
  - 物流摘要
  - 时间线
  - 地图点位
  - 当前状态

## 12. 顺丰 API 接入要点

### 12.1 公共请求模式

顺丰新 API 的公共请求通常包含以下核心字段：

- `partnerID`
- `requestID`
- `serviceCode`
- `timestamp`
- `msgData`
- `msgDigest`

实现要求：

- `requestID` 必须全局唯一
- `msgData` 为业务 JSON 串
- `msgDigest` 通过官方规则计算
- 所有请求统一由 `SfExpressClient` 发出

### 12.2 下单能力

在管理员发货时使用下单接口。

输入应包含：

- 订单号
- 寄件方信息
- 收件方信息
- 商品信息
- 运单附加信息

输出重点关注：

- 运单号
- 下单结果
- 错误码
- 错误信息

### 12.3 路由查询能力

轮询阶段的核心接口。

建议策略：

- 发货成功后立即触发一次路由查询
- 后续对运输中的单进行周期查询
- 对已签收单停止高频查询

### 12.4 路由注册能力

当前环境暂时无公网 HTTPS 地址，因此第一阶段不启用。

第二阶段具备公网地址后启用：

- 注册回调地址
- 接收顺丰路由推送
- 走与轮询完全相同的标准化和落库逻辑

## 13. 路由同步策略设计

### 13.1 第一阶段：轮询

适用条件：

- 当前没有公网可用 HTTPS 回调地址
- 需要尽快完成沙箱联调

建议轮询策略：

- 新发货单：每 10 分钟同步一次
- 运输中：每 30 分钟同步一次
- 接近签收：每 60 分钟同步一次
- 已签收：停止同步
- 超过最大天数：停止同步并进入归档状态

### 13.2 第二阶段：推送为主，轮询兜底

适用条件：

- 有公网 HTTPS 回调地址
- 可在安全配置中放行顺丰回调入口

建议策略：

- 顺丰推送到达后立即落库
- 定时轮询只处理以下异常情况：
  - 推送漏掉
  - 回调失败
  - 本地落库异常
  - 状态长期未更新

### 13.3 幂等策略

无论轮询还是推送，幂等必须统一。

建议幂等键：

```text
provider_code + tracking_no + event_time + event_code + content_hash
```

如果顺丰原始报文有稳定事件唯一标识，则优先使用外部事件 ID。

## 14. 状态映射设计

### 14.1 内部统一状态

项目内部建议继续使用统一状态：

- `CREATED`
- `SHIPPING`
- `DELIVERED`
- `EXCEPTION`

### 14.2 顺丰到内部状态映射

按节点文本和操作码做标准化映射：

- 已下单、待揽收：`CREATED`
- 已揽收、运输中、转运中、派送中：`SHIPPING`
- 已签收：`DELIVERED`
- 退回、派送异常、地址异常等：`EXCEPTION`

注意：

- 映射规则不能只依赖中文文案
- 需要优先依赖官方操作码或标准字段
- 文案匹配只作为兜底

## 15. 地图增强设计

### 15.1 设计原则

地图不是为了伪造真实运输路线，而是为了增强空间感知。

### 15.2 展示方式

推荐使用“城市跳点地图”：

- 起点：仓库或寄件地
- 中间点：顺丰节点地点对应的城市或站点
- 当前点：最新节点高亮
- 终点：收件地址所在城市

视觉形式建议：

- 轻量连线
- 当前节点呼吸动效
- 起终点不同图标
- 地图外增加状态摘要卡片

### 15.3 坐标来源

优先级如下：

1. 本地缓存命中坐标
2. 顺丰节点地点经过标准化后，调用高德地理编码获取坐标
3. 如果网点地址不够精确，则退化为城市级坐标
4. 如果仍无法解析，则不展示该点，只展示时间线

### 15.4 地图降级策略

如果节点无法转坐标：

- 地图只显示起点、当前点、终点
- 时间线保持完整
- 页面不报错

### 15.5 不推荐的做法

不建议：

- 用高德驾车路径去“拟合”顺丰真实运输路线
- 用平滑插值假装物流车辆连续移动
- 把城市级点位包装成精确轨迹

原因：

- 真实性差
- 误导用户
- 后续维护成本高

## 17. 错误处理与降级设计

### 17.1 下单失败

场景：

- 顺丰接口超时
- 鉴权失败
- 地址校验失败
- 沙箱配置错误

处理策略：

- 发货接口返回明确错误原因
- 不写入本地顺丰物流记录
- 不更新订单为已发货

### 17.2 路由查询失败

处理策略：

- 记录同步日志
- 增加重试次数
- 不覆盖已有成功节点
- 本地仍展示最后一次成功同步结果

### 17.3 地图增强失败

处理策略：

- 不影响时间线展示
- 标记 `geocode_status = FAILED`
- 后续允许定时补偿重试

### 17.4 推送启用后的回调失败

处理策略：

- 回调入库失败时记录原始报文
- 轮询任务兜底补偿
- 对外返回符合顺丰要求的应答格式

## 18. 安全设计

### 18.1 配置安全

- `client-code` 与 `check-word` 不入库
- 统一放私有配置或环境变量
- 日志打印时脱敏

### 18.2 回调安全

未来启用推送后：

- 新增顺丰回调专用接口
- 在安全配置中仅放行该接口
- 校验签名
- 校验来源格式
- 做幂等落库

### 18.3 日志安全

日志中避免打印：

- 完整收件人手机号
- 全量详细地址
- 密钥字段

## 22. 风险清单

### 22.1 顺丰节点地点文本不稳定

影响：

- 地理编码命中率可能不稳定

应对：

- 做地点标准化
- 做缓存
- 无法精确解析时退化为城市级坐标

### 22.2 当前项目存在模拟物流逻辑干扰

影响：

- 真实顺丰数据可能被本地模板轨迹覆盖或污染

应对：

- 引入 `provider_type`
- 严格隔离模拟物流与第三方物流逻辑

### 22.3 当前无公网回调

影响：

- 不能启用推送

应对：

- 先上轮询
- 代码结构预留回调能力

### 22.4 地图展示被误解为真实运输路线

影响：

- 用户认知偏差

应对：

- 明确采用“城市跳点展示”
- 不使用拟真连续路径

## 23. 结论

本项目适合接入顺丰沙箱，但必须遵循以下原则：

- 第三方物流与本地模拟物流隔离
- 真实路由节点为主数据源
- 时间线是真实主体，地图是优雅辅助
- 当前先轮询，未来升级推送
- 结构从一开始就按生产可切换设计

按本文方案实施后，可获得以下结果：

- 管理员发货可真实对接顺丰沙箱
- 本地保存真实运单号与真实路由
- 前端可优雅展示真实物流过程
- 后续切生产环境时无需推翻整体架构
